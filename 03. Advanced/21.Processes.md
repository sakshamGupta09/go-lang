
# Analogy

- Think of all programs on your OS as running restaurants.
- Goroutines are workers inside the same kitchen of one resturtant.
- Processes are completely separate restaurants.

## Goroutines

- When we run our program, a process is created and within that single process we create multiple goroutines(workers) that are working concurrently.
- All goroutines share the same memory and CPU resources. So, one gooroutine's bug can affect others. A crash in one of them can crash the whole process.

## Processes

- A process is an entirely separate running program which gets it's ownmemory, CPU, other resources from the OS.
- Your Go app runs which is one process and it can start a completely new program execution (process) in your OS.


## Spawning a process

- Ask the OS to run another independent program. One process is created when we run our GO program and anothe rone will be created when we ask our OS to run another program form our GO code.
- This is done using the os/exec package.

```go
func main() {
	cmd := exec.Command("echo", "Hello World!")
	output, err := cmd.Output()

	if err != nil {
		log.Fatal(err)
	}
	fmt.Println(string(output))
}
```
- Please create anothe rprocess than runs this command echo.


## Why use another process instead of goroutines?

Because sometimes you want:

- Isolation
- Safety
- Independent resources

## Concurrency

- Multiple processes run concurrently under the OS scheduler.
- But unlike goroutines, they do not share the memory.
- Two porcesses communicate via files, sockets, pipes, stdin/stdout
