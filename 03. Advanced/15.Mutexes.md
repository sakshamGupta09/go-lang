# Mutexes

## Shared Data + Multiple Goroutines

- We have some shared data and multiple goroutines are updating that data simultaneously.
- This can lead to race conditions.
- Ideally when one goroutine updates sahared data, next goroutine should work on the updated value.
- We lock the shared resource so that no other goroutine can access it. When we are done with the operations, we release the lock on shared resource.
- Multiple lines of code can be locked and unlocked using mutexes.
- Code written between lock and unlock is critical section.

```go
func main() {
	var wg sync.WaitGroup

	counter := 0

	for range 1000 {
		wg.Add(1)
		go func() {
			defer wg.Done()
			counter++
		}()
	}

	wg.Wait()
	fmt.Println(counter)
}
```
- Here output will keep varying.

## Mutex (Mutual Exclusion)

- Mutex ensures that only ONE goroutine can access this critical section at a time.
- Think of it like a bathroom lock, If someone is inside → others must wait. When they leave → next person enters.
- Always ensure to relese the lock using defer.


## Basic Example

```go
package main

import (
	"fmt"
	"sync"
)

func main() {
	var counter int
	var mu sync.Mutex
	var wg sync.WaitGroup

	for i := 0; i < 1000; i++ {
		wg.Add(1)

		go func() {
			defer wg.Done()

			mu.Lock()   // 🔒 lock
			counter++
			mu.Unlock() // 🔓 unlock
		}()
	}

	wg.Wait()
	fmt.Println(counter) // always 1000
}
```

## Structs

```go
type Counter struct {
	mu    sync.Mutex
	count int
}

func (c *Counter) IncrementCount() {
	c.mu.Lock()
	defer c.mu.Unlock()
	c.count++
}

func (c *Counter) ReadCount() int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.count
}

func main() {
	var wg sync.WaitGroup
	numGoroutines := 10000

	counter := &Counter{}

	for range numGoroutines {
		wg.Add(1)

		go func() {
			defer wg.Done()
			counter.IncrementCount()
		}()
	}

	wg.Wait()

	fmt.Println(counter.ReadCount())
}
```
