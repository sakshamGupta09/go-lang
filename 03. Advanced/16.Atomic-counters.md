# Atomic counters


## Problem

- Multiple goroutines updating a shared data leading to race conditions.
- Mutex can solve this problem but involves locking which can be heavy.


## Atomic Counters (Lock-Free and faster)

Instead of 
```go
counter++
```

You do:
```go
atomic.AddInt64(&counter, 1)
```

## Basic Example

```go
package main

import (
	"fmt"
	"sync"
	"sync/atomic"
)

func main() {
	var counter int64
	var wg sync.WaitGroup

	for i := 0; i < 1000; i++ {
		wg.Add(1)

		go func() {
			defer wg.Done()
			atomic.AddInt64(&counter, 1)
		}()
	}

	wg.Wait()
	fmt.Println("Final Counter:", counter)
}
```

## Atomic operations

### 1. Increment / Add

```go
atomic.AddInt64(&counter, 1)
```

### 2. Read Value Safely

```go
value := atomic.LoadInt64(&counter)
```

### 3. Set Value

```go
atomic.StoreInt64(&counter, 100)
```

### 4. Compare And Swap (CAS) 🔥

```go
atomic.CompareAndSwapInt64(&x, old, new)
```

- If x == old, then set x = new and return true else return false.
- Between your read and write, someone else may change the value. CAS helps you detect that change and retry safely.

```go
counter = 5
```
- Two goroutines try to increment using CAS.

🧵 Goroutine 1

```go
old = 5
new = 6
CAS(counter, 5, 6) → succeeds ✅
counter = 6
```

🧵 Goroutine 2 (at same time)

```go
old = 5
new = 6
CAS(counter, 5, 6) → fails ❌ (because now counter = 6)
```
- So G2 knows: “Oops, someone changed it. I need to retry.
