# sync.Cond

- Goroutines need to wait for some condition to become true.
- Multiple consumers will wait for a condition to be true, producer will enable the condition.
- A sync.Cond has 3 important methods and it always works with a mutex.

| Method        | Meaning                     |
| ------------- | --------------------------- |
| `Wait()`      | Sleep until notified        |
| `Signal()`    | Wake one waiting goroutine  |
| `Broadcast()` | Wake all waiting goroutines |

## Mental Model

```go
Mutex = protects shared state

Cond = notification system saying:
       "Hey, state changed.
        Wake up and check again."
```
- sync.Cond does NOT tell you what changed or whether condition is true.
- It only says:“Something may have changed.”

## Wait

Wait methods does three things -

- Unlocks the mutex. I am going to sleep now, but I will release teh lock before going. Other goroutines can now modify data.
- Puts goroutine to sleep. No CPU wasted.
- When signaled, wakes up, re-locks mutex automatically and continues execution.


## Why use for loop around Wait?

Always do

```go
for !condition {
    cond.Wait()
}
```
NOT

```go
if !condition {
    cond.Wait()
}
```

- Because waking up does NOT guarantee condition became true.
- So after waking up, check again and if condition still false → sleep again

## Basic Example

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	var mu sync.Mutex
	cond := sync.NewCond(&mu)

	ready := false

	// Waiting goroutine
	go func() {
		mu.Lock()

		for !ready {
			fmt.Println("Waiting...")
			cond.Wait()
		}

		fmt.Println("Received signal!")
		mu.Unlock()
	}()

	time.Sleep(2 * time.Second)

	// Producer goroutine
	mu.Lock()
	ready = true
	fmt.Println("Sending signal...")
	cond.Signal()
	mu.Unlock()

	time.Sleep(time.Second)
}
```

## Step-by-step intuition

```go
Goroutine:
    lock mutex
    check condition

    condition false?
        yes -> sleep using Wait()

Producer:
    lock mutex
    change shared state
    Signal()

Waiting goroutine:
    wakes up
    re-locks mutex
    checks condition again
```


## Producer Consumer

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func main() {
	var mu sync.Mutex
	cond := sync.NewCond(&mu)

	queue := []int{}

	// Consumer
	go func() {
		mu.Lock()

		for len(queue) == 0 {
			fmt.Println("Queue empty. Waiting...")
			cond.Wait()
		}

		item := queue[0]
		queue = queue[1:]

		fmt.Println("Consumed:", item)
		mu.Unlock()
	}()

	time.Sleep(2 * time.Second)

	// Producer
	mu.Lock()
	queue = append(queue, 42)
	fmt.Println("Produced: 42")

	cond.Signal()
	mu.Unlock()

	time.Sleep(time.Second)
}
```

## Signal vs Broadcast

- Signal wakes up one goroutine and asks it to recheck the condition.
- Broadcast Wakes up ALL waiting goroutines.
