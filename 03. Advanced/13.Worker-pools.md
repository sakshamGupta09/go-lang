# Worker Pool

## Analogy

Think of a worker pool like a restaurant kitchen.

- Customers place orders → these are your tasks/jobs
- The kitchen has 5 chefs → these are your workers
- Orders wait in a queue → this is usually a channel/queue
- Any free chef picks the next order and cooks it

Instead of creating a new chef for every order, you keep a fixed number of chefs working continuously.

## Need

- If we have a large number of tasks to perform, it's not ideal to create goroutine for every task. 10K tasks should not mean 10K goroutines.
- This can cause high memory usage, too much CPU context switching, database overload, API rate limits etc making overall system unstable.
- Instead we say, only 10 workers will work at a time.

## 1. A queue of jobs

- Tasks wait here.

```go
jobs := make(chan int)
```

## 2. Fixed number of workers

- Workers continuously take jobs from the queue. When a worker gets free they pick another job.

```go
for job := range jobs {
    // process job
}
```

## 3. Producer

- Something puts jobs into the queue.

```go
jobs <- 1
jobs <- 2
jobs <- 3
```

## Basic Example

```go
package main

import (
	"fmt"
	"time"
)

func worker(id int, jobs <-chan int) {
	for job := range jobs {
		fmt.Println("Worker", id, "processing job", job)

		time.Sleep(time.Second)

		fmt.Println("Worker", id, "finished job", job)
	}
}

func main() {
	jobs := make(chan int)

	// create 3 workers
	for i := 1; i <= 3; i++ {
		go worker(i, jobs)
	}

	// send 5 jobs
	for j := 1; j <= 5; j++ {
		jobs <- j
	}

	close(jobs)

	time.Sleep(time.Second * 6)
}
```
