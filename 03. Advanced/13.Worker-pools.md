# Worker Pools

- Creating too many goroutines is bad for CPU and memory.
- If we have a large number of tasks to perform, it's not ideal to create goroutine for every task. 10 K tasks should not mean 10K goroutines.
- A fixed number of goroutines (workers) that continuously pick tasks from a queue (channel).


## Simple Example

```go
package main

import (
	"fmt"
	"time"
)

// Worker function
func worker(id int, jobs <-chan int) {
	for job := range jobs {
		fmt.Printf("Worker %d processing job %d\n", id, job)
		time.Sleep(time.Second) // simulate work
	}
}

func main() {
	jobs := make(chan int)

	// Start 3 workers
	for i := 1; i <= 3; i++ {
		go worker(i, jobs)
	}

	// Send 5 jobs
	for j := 1; j <= 5; j++ {
		jobs <- j
	}

	close(jobs)

	time.Sleep(time.Second * 6)
}
```

1. Workers start first

```go
go worker(i, jobs)
```
- 3 goroutines (workers) waiting for the jobs.

2. Jobs are sent

```go
jobs <- j
```
- Workers pick them up automatically
- You don’t decide which worker gets what. Go scheduler handles that.
